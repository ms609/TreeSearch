# Plan: Architectural Design for Heuristic Search Tricks

## Goal

Design the shared infrastructure and build order for implementing
ratchet, drifting, tree fusing, sectorial search, and combined driven
search in C++. The aim is to identify the right abstractions upfront so
that implementing one trick doesn't force rework when the next one
arrives.

## Current state of the C++ engine

**Implemented (Phase 0–2 equivalent):**
- `DataSet`: bit-packed character blocks, pattern expansion by weight
- `TreeState`: flat topology arrays, per-node `prelim`/`final_` state sets
- Standard Fitch downpass + uppass (EW, no inapplicables yet)
- `local_reopt`-style indirect length calculation (`fitch_indirect_length`)
- SPR clip/regraft/unclip with incremental downpass + uppass
- NNI search loop (first-improvement hill-climbing)
- SPR search loop (first-improvement hill-climbing with full rescore)
- Rcpp bridge: `ts_fitch_score`, `ts_nni_search`, `ts_spr_search`

**Implemented (Step 1, 2026-03-13):**
- ✅ TBR search with flexible parameterized loop (ts_tbr.h/.cpp)
- ✅ from_above computation for efficient subtree rerooting
- ✅ Topology save/restore snapshots for safe undo
- ✅ ts_tbr_search() Rcpp bridge, 29 tests passing

**Not yet implemented (Phases 3–7 of core engine plan):**
- Union construct for bulk destination rejection
- Inapplicable-aware three-pass scoring
- Implied weights (per-character step extraction + IW scoring)
- Zero-length branch collapsing
- Split-based topology hashing
- Ratchet, drifting, tree fusing, sectorial search, driven search
- Wagner tree builder
- Parallelization

### The critical shared abstraction: the flexible TBR loop

Every trick modifies **how** TBR is run, not **what** TBR does:

| Trick | Acceptance rule | Stopping rule | Weight source |
|---|---|---|---|
| Standard TBR | Δ < 0 only | Convergence (no improvement in full pass) | Original |
| Ratchet (perturbation) | Δ ≤ 0 (equal OK) | After T/8 accepted changes | Perturbed weights |
| Ratchet (search) | Δ < 0 only | Convergence | Original |
| Drifting (suboptimal) | AFD/RFD criterion | After T/8 accepted changes | Original |
| Drifting (equal-score) | Δ ≤ 0 only | After T/8 accepted changes | Original |
| Cheap ratchet (perturbation) | Δ ≤ 0 | After T/8 changes | EW-only (skip IW/inapp) |
| SS internal search | Δ < 0 (or ≤ 0) | Convergence | Reduced dataset |

All of these are the same TBR loop with different parameters. **The TBR
search function must be parameterized from day one.** If we hardcode
standard acceptance for TBR and then have to refactor for ratchet/
drifting, that's exactly the rework we want to avoid.

### Proposed TBR search interface

```cpp
// Acceptance decision for a candidate rearrangement
enum class AcceptDecision { Accept, Reject };

// Parameters that control TBR search behavior
struct TBRParams {
    // Acceptance
    bool accept_equal = false;     // accept Δ = 0 moves?
    int afd_limit = 0;             // max absolute fit difference (0 = strict)
    double rfd_limit = 0.0;        // max relative fit difference (0 = strict)
    
    // Stopping
    int max_accepted_changes = 0;  // 0 = no limit (run to convergence)
    int max_clip_attempts = 0;     // 0 = no limit
    int max_hits = 1;              // consecutive equal-score hits before stopping
    
    // Weight modification (for ratchet)
    const std::vector<double>* block_weights = nullptr;  // null = use uniform
    bool cheap_mode = false;       // skip inapplicable/IW scoring
    
    // RFD tracking (for drifting)
    bool track_block_deltas = false;
};

struct TBRResult {
    int best_score;
    int n_accepted;
    int n_evaluated;
    bool converged;  // true if stopped due to no improvement
};

TBRResult tbr_search(TreeState& tree, const DataSet& ds,
                     const TBRParams& params);
```

This single function serves ratchet, drifting, standard search, and
SS internal search. The acceptance logic branches on params inside the
loop — the cost of a few conditionals per candidate is negligible
compared to the Fitch scoring.

**Risk assessment:** Is there a trick that needs something fundamentally
different from this parameterized loop? Reviewing all tricks:
- Ratchet: ✓ (perturbed weights + equal acceptance + early stop)
- Drifting: ✓ (AFD/RFD acceptance + early stop + block delta tracking)
- SS: ✓ (standard search on a different TreeState/DataSet)
- TF: Does NOT use TBR during exchange scoring. TF is a separate
  algorithm that swaps subtrees and scores the swap. But after TF finds
  improvements, a standard TBR pass follows.
- Driven search: Orchestrates calls to `tbr_search` with different params.

**Conclusion:** The parameterized TBR loop covers all use cases.
No fundamental rethinking needed.

## Design decision: double scores throughout (2026-03-13)

All result structs (`TBRResult`, `SearchResult`) and search loop score
variables now use `double` instead of `int`. This is a forward-compatibility
measure for implied weights (IW), where scores are `Σ k/(k+e)` (a double).

**What changed:**
- `TBRResult.best_score`: `int` → `double`
- `SearchResult.score`: `int` → `double`
- Search loop variables (`best_score`, `divided_length`, `candidate_score`,
  `actual`): `int` → `double`

**What stays `int`:**
- `fitch_downpass()`, `fitch_score()`, `fitch_indirect_length()` — these are
  EW-specific functions that genuinely return integer step counts. When IW
  arrives, a separate `iw_score()` function will return `double`.
- `fitch_incremental_downpass()` — returns an integer delta.

**Rationale:** For EW, int→double is lossless (all EW scores fit exactly in
double). For IW, the trick agents (ratchet, drifting, etc.) won't need a
score-type refactor — they're already building against `double` interfaces.

**Impact on inapplicable characters:** None. Inapplicable scoring still
produces integer step counts (EW). The indirect calculation becomes
approximate (not exact) for inapplicable blocks, but the verification
via `full_rescore()` — which TBR already does — handles this correctly.

## Structural interactions and risks

### 1. Sectorial search requires reentrant search

SS creates a `ReducedDataset` (smaller `DataSet` + `TreeState`) and
runs a full search on it. That search may itself include ratchet + TBR.
This means the search functions must not rely on global state — they
must operate on the passed `TreeState` and `DataSet` arguments.

**Current status:** The existing `spr_search` and `nni_search` already
take `TreeState&` and `const DataSet&` parameters, so they're already
reentrant. The TBR search should follow the same pattern. ✓ No rework
needed.

### 2. Tree fusing needs split infrastructure + subtree scores

TF identifies shared clades between trees via split intersection, then
swaps subtrees and scores the exchange. It needs:

(a) **Split computation per tree**: Compute the set of bipartitions.
    This is a straightforward postorder traversal building bitsets.
    
(b) **Shared group finding**: Intersect split sets of two trees. Fast
    with sorted vectors or hash sets.
    
(c) **Exchange scoring**: After swapping a subtree, walk root-ward
    recomputing downpass states and accumulating the score delta. This
    is similar to the incremental downpass after SPR clip, but instead
    of restoring one node, we're replacing a whole subtree.

(d) **Subtree score accounting**: To know whether an exchange helps, we
    need the current subtree score (sum of steps within the clade). This
    can be maintained incrementally or computed on demand from
    `local_cost[]` (sum of popcount over the subtree's internal nodes).

**Risk:** The exchange scoring (c) is the only novel operation. It's
architecturally similar to SPR regraft scoring but with a multi-node
subtree swap instead of a single-node reattachment. The incremental
downpass infrastructure from SPR should be reusable with minor
extension.

### 3. Drifting RFD needs per-block step deltas

The drifting acceptance criterion uses the Relative Fit Difference:
how much do step-increasing characters outweigh step-decreasing ones?
This requires tracking per-block step changes during move evaluation.

The current `fitch_indirect_length` returns only the total extra steps.
For drifting, we need it to also return per-block deltas (or at least
the sum of increases vs decreases).

**Solution:** Add a `BlockDeltas` output parameter to the indirect
length function, populated only when `track_block_deltas` is set. This
is a minor extension to the existing function, not a redesign.

### 4. Ratchet weight perturbation is block-level

With bit-packed scoring (64 chars per block), character-level weight
perturbation isn't practical. Instead:

- **Per-block weight array**: Replace uniform scoring with
  `block_weight[b] * popcount(needs_union)`. For standard search,
  all weights are 1. For ratchet, a few blocks get weight 0 or 2.
  
- **Per-character perturbation, block granularity**: Since characters
  within a block are expanded patterns (weight 1 each), "reweighting
  4% of characters" means zeroing ~4% of bits in `active_mask` for
  the perturbation phase. This is exact, not approximate.

**Implementation:** Create a `PerturbedWeights` struct that stores
modified `active_mask` values per block. The TBR loop checks
`params.block_weights` and uses them in scoring.

### 5. Zero-length collapsing interacts with the tree pool

Trees stored in the pool should be collapsed (zero-length branches
removed) to avoid storing trivially different dichotomous resolutions.
This means collapsing must be implemented before the pool is useful for
TF and driven search.

**However**: Collapsing is complex (needs final states from divided
tree + partial reoptimization). It can be deferred initially by
accepting some pool pollution (duplicate collapsed topologies stored
as distinct). The pool will still work, just less efficiently.

**Decision:** Implement pool without collapsing first. Add collapsing
as a refinement after the basic driven search works.

## Proposed build order

The order is driven by dependencies and by minimizing rework.

**Parallel agent strategy:** Steps 2, 3, 4, and 6 are independent of
each other (all depend only on Step 1). Each can be assigned to a
separate agent once Step 1 is complete. The plan notes launch points
below with "🚀 LAUNCH AGENT" markers. When the current agent completes
a step that unblocks parallel work, it should remind the user to launch
the parallel agents.

### Step 1: TBR search with flexible loop — ✅ COMPLETE (2026-03-13)

**Prerequisite:** None beyond current SPR implementation.

Extend SPR to TBR by adding subtree rerooting during the regraft
phase. Implement as `tbr_search()` with the `TBRParams` interface
from the start.

**Deliverables:**
- ✅ `ts_tbr.h` (32 lines): TBRParams, TBRResult structs, tbr_search() declaration
- ✅ `ts_tbr.cpp` (505 lines): Full TBR implementation with indirect calculation,
  from_above computation, save/restore topology snapshots, apply_tbr_move with
  path reversal rerooting, validate_topology for safety
- ✅ `ts_tbr_search()` Rcpp bridge in `ts_rcpp.cpp` + registration in `TreeSearch-init.c`
- ✅ `tests/testthat/test-ts-tbr-search.R` (210 lines): 29/29 tests passing
  - Valid structure, score consistency with TreeLength(), competitive with SPR,
    optimal on known small cases, parameter behavior, tree sizes 5–50 tips

**TBRParams interface (minimal, forward-compatible):**
```cpp
struct TBRParams {
    bool accept_equal = false;
    int max_accepted_changes = 0;  // 0 = no limit
    int max_hits = 1;
};
```
Future agents add fields (afd_limit, rfd_limit, block_weights, etc.) with
defaults — backward-compatible, no rework needed.

**Key bugs fixed:**
1. Segfault from tip sibling — start incremental downpass at nz (grandparent)
2. Wrong divided_length — subtract clipped node's local_cost
3. Topology corruption — snapshot-based undo instead of fragile unclip/re-clip

**See also:** `step1-tbr-implementation-spec.md` for full algorithm details.

**On completion:** 🚀 LAUNCH AGENTS for Steps 2, 3, 4, and 6 in
parallel (ratchet, drifting, pool+splits, Wagner tree). Each agent
should be given this plan file plus the relevant section of
`tricks-implementation-plan.md` for algorithmic detail.

**⚠️ READY FOR PARALLEL LAUNCH** — Step 1 is complete. Steps 2, 3, 4,
and 6 can proceed independently.

### Step 2: Ratchet

**Prerequisite:** Step 1 (flexible TBR).

Implement character perturbation and the ratchet cycle. Uses
`tbr_search()` with `accept_equal=true`, modified weights, and
`max_accepted_changes=T/8`.

**Deliverables:**
- `ts_ratchet.h/.cpp`: perturbation logic, ratchet cycle
- `PerturbedWeights` struct (modified active_masks)
- Integration with `TBRParams::block_weights`
- R interface: `ts_ratchet_search()`
- Tests: escapes local optima that plain TBR cannot

**Why second:** Simplest trick; validates the flexible TBR interface;
the R code already has the ratchet logic as a reference.

### Step 3: Tree drifting — ✅ COMPLETE (2026-03-13, Agent B)

**Prerequisite:** Step 1 (flexible TBR).

Implemented suboptimal-acceptance TBR with AFD/RFD criteria.
Used Option A (custom drift TBR loop) to avoid modifying shared
`ts_tbr.h/cpp`. RFD computed at block level via `local_cost[]`
comparison before/after move application.

**Deliverables:**
- ✅ `ts_drift.h` (38 lines): DriftParams, DriftResult, drift_search()
- ✅ `ts_drift.cpp` (~580 lines): drift_phase() with AFD/RFD acceptance,
  drift_search() with alternating suboptimal/equal-score cycles,
  all TBR helpers duplicated with drift_ prefix (no shared code modified)
- ✅ `ts_drift_search()` Rcpp bridge (9 params) + TreeSearch-init.c registration
- ✅ `tests/testthat/test-ts-drift-search.R`: 8 tests covering structure,
  score verification, no-worsen guarantee, AFD=0 conservatism, RFD limit
  effect, single cycle, various sizes (10/20/50), local-optima escape
- ✅ Compiles with zero warnings (`g++ -Wall -pedantic`)

**Design choices:**
- Custom drift TBR loop (Option A) rather than extending TBRParams,
  to avoid conflicts with other agents
- RFD computed by restore→rescore→re-apply→rescore cycle (expensive
  but correct; only triggered for suboptimal candidates within AFD limit)
- Equal-score drift cycles use existing `tbr_search()` with
  `accept_equal=true, max_accepted_changes=T/8`

**Not yet integration-tested** — full package link blocked by
pre-existing linker error in ts_wagner.cpp (Agent D).

### Step 4: Split hashing + tree pool

**Prerequisite:** None beyond basic `TreeState`.

Implement split computation, hashing, and pool management. This is
pure infrastructure with no dependence on specific tricks.

**Deliverables:**
- `ts_splits.h/.cpp`: split computation from `TreeState`, split hashing
- `ts_pool.h/.cpp`: `TreePool` with hash-based dedup, score-based
  eviction
- Tests: dedup correctly identifies identical topologies; pool evicts
  worst trees

**Why here:** Tree fusing (Step 5) and driven search (Step 7) both
need the pool. Building it now keeps the dependency chain clean.

### Step 5: Tree fusing

**Prerequisite:** Step 4 (split hash + pool).

Implement shared-group identification and exchange scoring.

**Deliverables:**
- `ts_fuse.h/.cpp`: shared group finding via split intersection,
  exchange scoring (root-ward walk), bottom-up ordering
- Equal-score exchange option
- R interface: `ts_tree_fuse()`
- Tests: fusing 5 independent RAS+TBR trees beats any single one

**Why here:** TF is simpler than SS and dramatically improves
multi-replicate searches. Having TF before SS also means SS sectors
can use TF internally for large sectors.

**Prerequisite check:** Steps 2, 3, and 6 should also be complete (or
nearly so) before proceeding to Step 7. If parallel agents are still
running, wait for them or proceed with Step 7 using only TBR + TF.

### Step 6: Wagner tree

**Prerequisite:** Step 1 (TBR for the search that follows Wagner).

Implement C++ Wagner tree builder with indirect scoring for
insertion positions.

**Deliverables:**
- `ts_wagner.h/.cpp`: greedy addition with `local_reopt`-style scoring
- Random addition sequence support
- R interface: `ts_wagner_tree()`
- Tests: scores match R `AdditionTree()`

**Why here:** Needed by driven search (Step 7) and by SS (for RAS
within sectors). Could be done earlier but isn't blocking Steps 2–5.

### Step 7: Sectorial search

**Prerequisite:** Steps 1 (TBR), 6 (Wagner). Benefits from Steps 2
(ratchet), 5 (TF) for internal sector search.

This is the most architecturally complex trick. Implement reduced
dataset construction, sector selection, and recursive search.

**Deliverables:**
- `ts_sector.h/.cpp`: `ReducedDataset` construction with HTU state
  sets, RSS and XSS sector selection, sector reinsertion with partial
  rescore
- Character filtering for reduced datasets
- Recursive search on sectors (reuses `tbr_search` + `ratchet`)
- R interface: `ts_sectorial_search()`
- Tests: SS outperforms plain TBR on 100+ taxon datasets

**Why last among individual tricks:** Most complex; needs downpass
state extraction at arbitrary nodes; benefits from all other tricks
being available for sector-internal search.

### Step 8: Combined driven search

**Prerequisite:** All of Steps 1–7.

Orchestrate all tricks in a combined search loop with convergence
criteria.

**Deliverables:**
- `ts_driven.h/.cpp`: driven search loop, combosearch structure for
  large trees, MSD-based adaptive stopping
- Consensus stability checking (optional)
- `MaximizeParsimony2()` R wrapper
- Progress reporting + interrupt checking
- Benchmark suite vs existing `MaximizeParsimony()`

### Step 9 (deferred): Zero-length collapsing

Add collapsing to improve pool efficiency and search effectiveness.

### Step 10 (deferred): Parallelization

Add replicate-level parallelism with thread-safe pool.

## Key interfaces summary

The following interfaces are designed to be stable across all steps:

```
┌─────────────────────────────────────────────────────┐
│  driven_search()  (Step 8)                          │
│    Orchestrates all tricks                          │
├──────────┬──────────┬──────────┬───────────────────┤
│ ratchet()│ drift()  │tree_fuse()│sectorial_search()│
│ (Step 2) │ (Step 3) │ (Step 5)  │ (Step 7)         │
├──────────┴──────────┴──────────┴───────────────────┤
│  tbr_search(tree, data, TBRParams)  (Step 1)       │
│    — configurable acceptance, stopping, weights     │
├─────────────────────────────────────────────────────┤
│  TreePool + SplitHash  (Step 4)                     │
├─────────────────────────────────────────────────────┤
│  TreeState + DataSet + Fitch scoring  (existing)    │
└─────────────────────────────────────────────────────┘
```

## What can be built in parallel (agent assignment)

After Step 1 (TBR), four independent workstreams can proceed
concurrently, each assigned to a separate agent:

```
                    ┌─ Agent A: Step 2 (Ratchet) ──────────┐
                    │                                       │
Step 1: TBR ────────┼─ Agent B: Step 3 (Drifting) ─────────┤
(this agent)        │                                       ├─→ Step 5 (TF)
                    ├─ Agent C: Step 4 (Pool + Splits) ─────┤   then
                    │                                       │   Step 7 (SS)
                    └─ Agent D: Step 6 (Wagner tree) ───────┘   then
                                                                Step 8 (Driven)
```

Each parallel agent needs:
- This architectural plan (for interface contracts)
- The relevant section of `tricks-implementation-plan.md`
- Read access to `ts_tree.h`, `ts_fitch.h`, `ts_data.h`, `ts_search.h`
  (for existing interfaces)
- The `TBRParams`/`TBRResult` interface from Step 1 (once complete)

Steps 5, 7, and 8 are sequential and depend on the parallel steps
converging. They can be handled by this agent or a new one.

## Relationship to existing plans

This plan does **not** replace the existing `tricks-implementation-plan.md`
(which contains the algorithmic detail for each trick) or the
`tree-search-engine-plan.md` (which specifies the core engine). It sits
between them as an **architectural bridge**, specifying:

1. The shared abstractions that prevent rework
2. The build order that respects dependencies
3. The key interfaces that multiple tricks share

The algorithmic specifications in `tricks-implementation-plan.md`
(SS sector selection, TF exchange scoring, ratchet parameters, drifting
RFD criteria, etc.) remain the reference for implementation details.
